# SageMaker HyperPod CLI

A Python **CLI + SDK** for managing Amazon SageMaker HyperPod clusters orchestrated by **Amazon EKS/Kubernetes**. The CLI binary is `hyp`. It abstracts away direct Kubernetes/CloudFormation work for the full lifecycle of training jobs, inference endpoints, cluster infrastructure, and interactive dev spaces.

## The Five Domains

| Domain | SDK class(es) | CLI noun(s) | Backed by |
|--------|---------------|-------------|-----------|
| Training | `HyperPodPytorchJob` | `hyp-pytorch-job`, `hyp-recipe-job` | K8s CRD `HyperPodPyTorchJob` (group `sagemaker.amazonaws.com/v1`) |
| Inference (JumpStart) | `HPJumpStartEndpoint` | `hyp-jumpstart-endpoint` | K8s CRD (`JumpStartModel`) |
| Inference (custom) | `HPEndpoint` | `hyp-custom-endpoint` | K8s CRD `InferenceEndpointConfig` |
| Cluster infra | `HpClusterStack` | `cluster-stack`, `cluster` | CloudFormation + Helm |
| Dev spaces | `HPSpace`, `HPSpaceTemplate` | `hyp-space`, `hyp-space-template`, `hyp-space-access` | K8s CRD (group `workspace.jupyter.org` family) |

## Core Architectural Pattern: Schema-Driven CLI

The single most important concept. Each domain's config schema lives in a **separate, independently-versioned pip package** at the repo root (`hyperpod-*-template/`). Each exposes `SCHEMA_REGISTRY` (version → flat Pydantic model) and `TEMPLATE_REGISTRY` (version → Jinja2 string) via its `registry.py`. The CLI generates its flags **dynamically** from those schemas.

```
CLI flags ──▶ flat Pydantic model ──▶ .to_domain() ──▶ SDK domain object ──▶ .create()
   (generated by                                                              │
    @generate_click_command)                                                  ▼
                                                          K8s CustomObjectsApi (CRD)  OR  CloudFormation (CFN)
```

Central wiring of noun → registry → schema type is in `src/sagemaker/hyperpod/cli/constants/init_constants.py::TEMPLATES`. See `templates-and-init.md`.
